    Build Log

    This is the real development history of gitmind — every commit, what broke, what we learned, and what gitmind said about itself as it was being built.

    gitmind was built in a single session and tested on its own repository from commit 2 onward. The metadata below is real — generated by the tool analyzing its own diffs.

    Commit 1 — Initial scaffold

    Hash: 6e78acd

    feat: initial gitmind engine — diff reader, LLM analyzer, metadata writer, CLI query tool, install hook

    Files created in one shot:

    • core/diff_reader.py — reads git diff, changed files, commit message, hash
    • core/llm.py — Ollama interface with structured prompt
    • core/metadata.py — load/save/update metadata.json
    • core/engine.py — orchestrates the pipeline
    • hooks/post-commit — bash trigger
    • hooks/install.sh — installs gitmind into any repo
    • cli/query.py — features, files, history, stale

    Hook result: No diff found — skipping metadata update.

    Why: This was the root commit. git diff HEAD~1 HEAD has no parent to diff against on the first commit, so it returns nothing. The hook ran but silently skipped.

    Commit 2 — First-commit edge case fix

    Hash: ffaff32

    fix: handle first commit diff, add .gitignore to exclude venv

    Two problems found during commit 1:

    1. Root commit has no HEAD~1 — fixed with _is_first_commit() guard, falling back to git show HEAD
    2. The venv directory (700+ files) got committed because .gitignore didn't exist yet

    Hook result: JSON parse error

    JSONDecodeError: Expecting property name enclosed in double quotes: line 4 column 280
Raw output: {
    "what_changed": [".gitignore", ".env"],

    What happened: The LLM returned what_changed as an array instead of a string. The prompt said "one sentence" but the model ignored the type constraint. Also found // comments inside the JSON block — invalid JSON syntax.

    Commit 3 — Robust JSON parsing

    Hash: 2f91e1d

    fix: robust JSON parsing — extract, coerce field types, sanitize feature_name

    Added three layers of defense:

    • _extract_json() — strips markdown fences (```json ```), finds the first {...} block by bracket depth
    • _coerce() — converts wrong field types (list → string), fills files_touched from git if LLM left it empty, sanitizes feature_name to snake_case
    • Feature name sanitization: re.sub(r"[^a-z0-9_]", "_", name)

    Hook result: Still failing — different JSON error.

    The model was still outputting commentary after the closing } bracket, and using // comments inside the object. _extract_json() correctly isolated the object but the // comments made json.loads fail.

    Commit 4 — format: "json" enforcement

    Hash: 08fe52f

    fix: use ollama format:json to enforce valid JSON output from model

    Added one line to the Ollama API request:

    "format": "json"

    This constrains the model's output tokenizer to only emit valid JSON tokens. It cannot produce // comments, trailing text, or markdown fences when this is set.

    Hook result: First successful end-to-end run.

    [gitmind] Analyzing commit...
[gitmind] Feature tracked: _use_json__is_used_as_it_might_be_beneficial_for_handling...
[gitmind] The git diff has been analyzed and returned in JSON format.

    Feature name was 90+ characters — the coercion was working but the truncation logic had a word-boundary bug that produced an underscore-prefixed name.

    Commit 7 — Docs deployment fix

    Hash: 5906386

    fix: add contents:write permission and git config for docs deploy bot

    The docs workflow failed on first run:

    remote: Permission to chanakyavasantha/gitmind.git denied to github-actions[bot].

    The GitHub Actions bot needs explicit permission to push to the gh-pages branch. Fixed by adding to the workflow:

    Key Lessons From Building This

    1. format: "json" is non-negotiable with deepseek-coder Without it, the model adds // comments, trailing prose, and wraps output in markdown fences. With it, output is clean every time.

    2. Type coercion matters The model sometimes returns a list where a string is expected (e.g. "what_changed": [".gitignore", ".env"]). A _coerce() layer that fixes field types is more reliable than trying to write a prompt that prevents it.

    3. Large diffs confuse feature naming When 28 unrelated files change in one commit, the LLM doesn't have a clean signal for what the "feature" is. Phase 2 will chunk diffs by directory and summarize per-group before naming.

    4. Test the tool on itself first Every bug surfaced in commits 2–4 was found because gitmind was running on its own repo. Dogfooding caught all the real edge cases faster than any test suite would have.
